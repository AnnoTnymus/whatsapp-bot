Anatomía del agente

Un orquestador central coordina 4 subagentes especializados, cada uno con su propio prompt y contexto. El orquestador decide a quién derivar según la intención del usuario.

🧠 Orquestador (Claude Opus 4.7)
Lee mensaje + historial + knowledge base → decide tono, flujo, derivación. Emite markers especiales: [[AFILIAR]] · [[SKILL:nombre]]
↓ deriva según intención ↓
⚖️ legal_faq
Ley 27.350, fallo Arriola, marco legal AR
📝 reprocann_guide
Trámite paso a paso, requisitos, costos
🌱 genetics_expert
Catálogo de cepas, efectos, terpenos
🆔 parseUserName
Extrae apodo + nombre completo
👁️ Vision (DNI / REPROCANN)
Lee documentos, extrae campos
🔍 QA Agent (on-demand)
/admin/qa-report audita conversaciones

¿Cómo decide el orquestador?

El orquestador no "llama funciones": emite markers de texto que el runtime detecta con regex. Simple, determinista, auditable.

1

Mensaje entra

GreenAPI webhook → withChatLock serializa por chatId. Distintos números corren en paralelo.

→
2

Orquestador razona

Claude lee mensaje + últimos 8 turnos + knowledge base. Elige tono (cordial vs off-topic). Decide si necesita un experto.

→
3

Emite marker si aplica

Al final de su reply agrega [[SKILL:legal_faq]] o [[AFILIAR]]. Si no, responde directo.

4

Runtime intercepta

parseSkillMarker(reply) detecta el marker → invoca skill con un prompt especializado + mismo historial.

→
5

Respuesta al usuario

La skill reemplaza la reply del orquestador. El marker se strippea. El usuario nunca ve el protocolo.

💡 Ventaja: cada skill tiene contexto focalizado (500-1000 tokens) en vez de un system prompt gigante mezclando todo.

Las 3 skills especializadas

[[SKILL:legal_faq]]

⚖️ Legal FAQ

Experto en marco legal del cannabis en Argentina. Conoce Ley 27.350, Decreto 883/2020, Ley 27.669, fallo Arriola, figura de asociación civil para clubes.

Se activa con: "¿es legal?", "me pararon", "qué dice la ley sobre..."
[[SKILL:reprocann_guide]]

📝 REPROCANN Guide

Guía paso a paso del trámite. Requisitos, patologías aprobadas, modalidades (autocultivo / solidario / farmacia), tiempos, costos, renovación.

Se activa con: "cómo saco el reprocann", "qué necesito", "cuánto demora"
[[SKILL:genetics_expert]]

🌱 Genetics Expert

Asesor de cepas según efecto buscado. Catálogo del club: indicas, sativas, híbridas. Conoce perfiles de THC/CBD, terpenos, tolerancia.

Se activa con: "qué me recomendás", "quiero algo para dormir", "cepas para..."

Cada skill tiene su propio system prompt de ~500-1000 tokens. El orquestador solo sabe cuándo invocarlas, no necesita "saber de genéticas" él mismo.

Audio + personalidad

Notas de voz transcritas en segundos. Stickers y emojis respondidos con humor rioplatense, no con "no entiendo".

1

Audio → Edge Function

msgType=audioMessage + downloadUrl → Supabase whatsapp-audio-stt con STT_SHARED_SECRET.

2

Deepgram ES en ~1-2s

Validación de hostname contra allowlist, POST con language=es. Si falla → el bot pide escribirlo.

Pipeline reusado

Off-topic con rotación

Capacidades del sistema

💬

Atención al cliente primero

Prioriza responder dudas (horarios, precios, productos). Solo entra al flujo de afiliación cuando el usuario lo pide explícitamente.

🎯

Detección de intención

Marker [[AFILIAR]] cuando el usuario quiere anotarse. Regex wantHuman detecta pedido de atención humana → email al admin.

🆔

Parseo inteligente del nombre

"Martin pero me dicen Tincho" → apodo=Tincho, completo=Martin. Pide aclaración si es ambiguo. CRM guarda nombre completo, saludos usan apodo.

👁️

Vision de documentos

Claude analiza DNI y REPROCANN. Si no puede leer un dato, lo pide por texto en vez de rechazar la foto.

⏰

Follow-ups automáticos

Recontacta según motivo: sin REPROCANN, tramitando, docs incompletos, inactivo. Cron configurable en minutos o días.

🎭

Respuestas off-topic variadas

Stickers, audios, emoji-only: respuestas jocosas con rotación anti-repetición por chat (memoria de última variante usada).

⚡

Concurrencia por chatId

Lock por chat evita race conditions. Chats distintos procesan en paralelo. Contador inFlightWebhooks en /health.

🔔

Alertas al admin

Email vía Resend cuando: usuario pide atención humana, cuota GreenAPI agotada (HTTP 466), error crítico.

🔍

QA Agent on-demand

Endpoint /admin/qa-report: audita últimas N conversaciones contra una rúbrica (tono, precisión, cierre). Solo lectura.

🎙️ Audio → Texto (Deepgram)

Los usuarios mandan notas de voz. El bot las transcribe en segundos y las procesa como si fueran texto — sin pedirles que repitan por escrito.

1

Audio entra

GreenAPI webhook con msgType=audioMessage + downloadUrl (hospedado en DO Spaces de GreenAPI).

→
2

Edge Function (Deno)

Bot llama a whatsapp-audio-stt en Supabase con x-stt-secret. La función valida el hostname del URL contra una allowlist.

→
3

Deepgram ES

POST con language=es. Transcript vuelve en ~1-2 s. Si falla → el bot pide escribirlo.

💡 El transcript entra al mismo pipeline que un mensaje de texto — orquestador, skills, state machine, todo reusado. No hay rama especial.

🔒 Seguridad end-to-end

Hardening por capa: cada superficie expuesta tiene su propio secret, y las comparaciones son timingSafeEqual para evitar leaks.

🔑

Webhook firmado

WEBHOOK_SECRET obligatorio en prod. GreenAPI lo envía como Authorization: Bearer. Acepta también query ?token= y header x-webhook-secret.

🛡️

Admin & test routes

ADMIN_API_TOKEN gate en /admin/* y /test/*. ENABLE_TEST_ROUTES=false en prod apaga todo el namespace de test.

🎙️

STT con secret propio

La Edge Function valida STT_SHARED_SECRET + STT_ALLOWED_HOSTS (solo descarga audios del CDN oficial de GreenAPI).

💾

Supabase service_role only

El bot nunca usa anon. RLS habilitado en las 4 tablas — clientes públicos no pueden escribir CRM ni historial.

⏱️

Rate limit por chat

30 mensajes/hora por chat_id. Evita abuso accidental y spam. Reset automático por ventana deslizante.

📧

Alertas proactivas

Email al admin cuando: usuario pide humano, cuota GreenAPI exhausta (HTTP 466), error crítico. Throttling 1/hora para no spamear.

Base de datos

4 tablas en Supabase con RLS habilitado y service_role bypass para el bot

👥

members

CRM principal. chat_id, nombre, estado_afiliacion, reprocann_status, fecha_alta, último contacto.

💬

conversation_history

Historial completo. Últimos 8 mensajes por chat, texto + timestamp. Base del contexto que se pasa al orquestador.

🔔

patient_followups

Cola de seguimientos. motivo, próxima notificación, intento actual, status. Cron procesa pendientes.

📊

patient_state

Estado conversacional. chat_id + step + documentos + collected_data (JSON con nombre_completo, datos del DNI, etc.).
